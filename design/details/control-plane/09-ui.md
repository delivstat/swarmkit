# 09 — UI

Scope: `packages/ui/` — Next.js 15 (app router) + React 19 + Tailwind 4 + Biome. Single-instance,
zero-auth today. `packages/ui/CLAUDE.md` describes three intended surfaces; only part is built.

## Pages (`app/`) and the serve coupling

| Route | Shows | serve endpoints (`lib/api.ts`) |
|---|---|---|
| `/dashboard` | health, workspace counts, recent jobs, canary summary | `health`, `validate`, `jobs`, `canary` |
| `/composer` | topology editor — 3 views + YAML | `topologies`, `topologyDetail`, `topologyYaml`, `saveTopology`, `reloadWorkspace` |
| `/topologies` | list + run dialog | `topologies`, `run` |
| `/skills`, `/archetypes` | catalog + YAML-in-dialog editor | `skills`/`archetypes` + `/api/*/{id}/yaml`, `saveSkill`/`saveArchetype` |
| `/jobs`, `/jobs/[id]` | job table + detail with SSE stream | `jobs`, `job`, `jobStreamUrl` |
| `/chat` | multi-turn conversation (SSE), traces/usage panels | `conversations`, `createConversation`, `sendMessageStream` |
| `/triggers` | read-only trigger table | `triggers` |
| `/canary` | per-topology versions + promote/rollback | `canary`, `canaryPromote`, `canaryRollback` |

**API client:** `lib/api.ts`, base URL `process.env.NEXT_PUBLIC_SWARMKIT_API ?? http://localhost:8000`.
Plain REST GET/POST + EventSource for SSE. **No auth headers, no 401 handling, no instance
selection.** Zustand is a dependency but unused (state via React hooks + `usePoll`).

## Composer reality

YAML-editing only: a read-only agent tree (Structure/Relationships/Network are visualizations of the
same tree) + a `<textarea>` for the whole-topology YAML (PropertyPanel toggle + bottom panel),
`saveTopology` → validate → reload. "New" creates a one-agent skeleton from a template. **No
form-based agent editing, no drag-drop, no conversational authoring.** (Confirmed in
[the composer read]; matches the user-facing assessment given earlier.)

## Control-plane implications

- **Reusable:** the REST-over-env-base-URL client, modular per-feature pages, no hardcoded instance
  ids. A control panel can build on this.
- **Needed for multi-instance:** auth-header injection + token/session handling + global 401→login;
  an instance/workspace **selector** (which instance to query); SSE routing per instance; a user
  profile/logout; optional RBAC-aware UI gating.
- **Decision:** extend this Next.js app into the control panel vs start a dedicated app. Given the
  OSS/commercial boundary (the polished multi-instance panel is Rynko's surface per
  `product-architecture.md`), the likely split is: keep this as the **single-instance OSS dashboard**,
  and build the fleet panel separately (reusing `lib/api.ts` patterns + the schema package).
- The **conversational authoring** the composer lacks is the surface most worth building in the
  panel (front the authoring swarm via serve), rather than more YAML-textarea editing.
