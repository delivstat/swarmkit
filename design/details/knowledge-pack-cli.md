# `swael knowledge-pack` CLI

## Goal

One command that produces a self-contained, paste-ready prompt an LLM
can answer Swael questions from. Operationalises the "docs are
consumed primarily by LLMs" principle from
[llm-friendly-knowledge.md](../../docs/notes/llm-friendly-knowledge.md)
and the "escape hatch for the 'I'm stuck — ask an LLM' flow" clause in
the Milestone 1 plan.

```
swael knowledge-pack                    # → stdout, no workspace overlay
swael knowledge-pack my/workspace       # → stdout, includes workspace + validation
swael knowledge-pack -o pack.md         # → pack.md
```

## Non-goals

- **Smart slicing.** Not selecting "just the relevant docs" — the user's
  LLM does that. The command ships the whole corpus, unabridged. Smart
  slicing belongs in the (future) knowledge MCP server (task #25), not
  here.
- **Fetching from GitHub at runtime.** The pack is a snapshot of local
  files. No network calls. Offline-usable.
- **Arbitrary query / chat.** `swael ask` (task #36) is the
  interactive observer. `knowledge-pack` is the one-shot "give me
  everything" flow.

## What's in the pack

Auto-discovered so new docs land without editing the CLI:

| Category | Source |
| --- | --- |
| Project overview | `README.md`, root `CLAUDE.md` |
| Canonical index | `llms.txt` |
| Authoritative design | `design/Swael-Design-v0.6.md`, `design/IMPLEMENTATION-PLAN.md` |
| Per-feature design notes | every `design/details/*.md` (excluding `README.md` and `_template.md`) |
| Cross-cutting notes | every `docs/notes/*.md` (excluding `README.md`) |
| Canonical schemas | every `packages/schema/schemas/*.json` |
| Schema fixtures | every `packages/schema/tests/fixtures/**/*.yaml` — both valid and invalid. Invalid fixtures double as "what the errors look like" examples |
| Per-package invariants | every `packages/*/CLAUDE.md` |

File order inside each category is sorted alphabetically for
reproducibility.

### When a workspace path is given

Appended after the corpus, in this order:
1. The workspace's `workspace.yaml`, plus every file under
   `topologies/`, `archetypes/`, `skills/`, `triggers/`, `schedules/`.
2. The output of `swael validate <path> --no-color` (success summary
   or error block). Captured verbatim so a reader sees what the user
   sees.

If the workspace is invalid, validation output is included but doesn't
fail the pack — the whole point is "I'm stuck, help me understand."

## Output shape

Single markdown document. Structure:

```markdown
# Swael Knowledge Pack

> Generated by `swael knowledge-pack` on 2026-04-22T10:45:00Z
> Includes: 38 files, 194 KB. Workspace overlay: hello-swarm.

## About this pack

You are an LLM ...<orientation paragraph>...

---

## Project overview

### `README.md`

<content>

### `CLAUDE.md`

<content>

---

## Authoritative design
... etc ...
```

- Each file gets an `h3` section with its repo-relative path.
- Markdown files are included as-is (raw markdown). Nested headings are
  fine for LLM readers — human readers are not the primary audience.
- JSON/YAML files are wrapped in triple-backtick fences with language
  tags.

## Resolution strategy

The CLI finds the corpus relative to the `swael_runtime` package
source: walk up until a directory contains both `CLAUDE.md` and
`design/`. Works for editable installs (`uv sync`) and source
checkouts — the current deployment model pre-v1.0.

**Known limitation for v1.0 publish:** once `swael` is installed from
PyPI without a source checkout, the corpus won't be on disk. Two options
when we get there:

1. Ship the corpus as package data (preferred — self-contained).
2. Fall back to fetching the `main` branch tarball (network, reproducibility
   concerns).

Neither is scoped for this PR. When the corpus isn't found, the CLI
exits with `2` and a message telling the user they're not in a source
checkout.

## Flags + exit codes

```
swael knowledge-pack [PATH] [-o FILE] [--no-fixtures]
```

- `PATH` (positional, optional): workspace directory. Triggers the
  workspace overlay.
- `-o, --output FILE`: write pack to file (overwrites). Default stdout.
- `--no-fixtures`: skip the schema fixtures section (~smaller pack).

Exit codes:
- `0` — pack written successfully (even if the workspace overlay had
  validation errors; those are *part* of the pack, not a CLI failure).
- `2` — corpus not found (not in a source checkout), or the requested
  workspace path doesn't exist.

## Test plan

`packages/runtime/tests/test_knowledge_pack.py`:

1. **Corpus shape.** Given a minimal faked corpus (temp dir with a
   `CLAUDE.md`, `design/`, a couple of notes), the generator produces
   all expected sections. Asserts section headers and file inclusion.
2. **Workspace overlay.** When called with a workspace path, the pack
   contains the workspace files and a validation block. Use the
   `examples/hello-swarm/workspace` example as input.
3. **Broken workspace overlay.** Called against
   `examples/hello-swarm/workspace-broken`, the pack includes the error
   block and the CLI still exits `0`.
4. **Output-to-file.** `-o FILE` writes to the file and produces no
   stdout.
5. **`--no-fixtures`.** Fixtures section is absent.
6. **No source checkout.** When the corpus resolver can't find a repo
   root, the CLI exits `2` with a clear message.

## Demo

`just demo-knowledge-pack` runs:

1. `swael knowledge-pack examples/hello-swarm/workspace | wc -c` —
   prints the pack size so the reader sees the order of magnitude
   ("it's bytes, not megabytes").
2. `swael knowledge-pack examples/hello-swarm/workspace-broken |
   head -60` — shows the pack header + "about this pack" orientation
   paragraph so a reader understands the shape without having to run
   it themselves.
